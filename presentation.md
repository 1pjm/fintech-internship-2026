# 전세대출 추천 스코어링 파이프라인 발표

---

## ① 어떤 기준으로 주제(도메인)를 잡았나

### 고른 도메인: **전세대출 (전세자금대출)**

### 왜 이 도메인인가

전세대출은 적금·카드와 달리 **"고객이 이미 계약을 체결한 뒤 신청"** 하는 구조다.
즉 서비스 실패(미승인·한도 부족)의 비용이 매우 크고, 고객은 선택지가 제한된 상태에서
"지금 당장 나에게 맞는 상품"을 빠르게 알아야 한다.

| 이유 | 설명 |
|------|------|
| **상환능력 피처가 풍부** | 연소득·DSR·신용점수·부채총액 등 스코어링에 쓸 수 있는 정량 데이터가 많음 |
| **정부 상품 + 시장 상품 혼재** | A1~A5 버팀목 계열(저금리 정부지원)과 은행·저축은행 56개 상품이 공존 → 고객별 최적 조합 추천 가치가 높음 |
| **Gate 설계가 명확** | 무주택·세대주·소득·DSR 등 법령·상품 약관에서 Hard Filter 조건을 그대로 가져올 수 있음 |
| **도메인 데이터 공개** | 금감원 금융상품통합비교공시 API로 실제 은행 상품(금리·한도·보증기관)을 수집 가능 |

> 고객 × 상품 조합(cross join) 구조를 채택해 **"고객 1명에게 최대 60개 상품을 동시 평가"** 하는 방식이 적금 실습과 가장 다른 점.

---

## ② 어떤 기준으로 스코어링을 했나

### 제외(Gate) 조건 — JeonseLoanGate

하나라도 0이면 해당 상품은 추천에서 **완전 제외** (곱 연산).

```
JeonseLoanGate(고객, 상품) =
    연령조건충족
  × 무주택조건충족
  × 세대주조건충족
  × 소득조건충족
  × 신용점수기준충족
  × DSR기준이하
  × 보증금한도충족
  × 주택유형충족
  × 전용면적충족
  × 중복대출없음
  × 정보완전성충족
  × 상품별필수조건충족
```

| Gate 항목 | 판정 기준 |
|-----------|----------|
| 연령 | 만 나이 ∈ [상품별 허용 범위] |
| 무주택 | 세대원 전원 무주택 = Y |
| 세대주 | 세대주 또는 예비세대주 = Y |
| 소득 | 부부합산 연소득 ≤ 상품별 소득상한 |
| 신용점수 | KCB ≥ 상품별 최저 기준 (정부 600점, 은행 650점) |
| DSR | 예상DSR ≤ 금융권역별 한도 (은행 40%, 2금융권 50%) |
| 보증금한도 | 전세보증금 ≤ 상품별 상한 (수도권/비수도권 차등) |
| 주택유형 | 임차유형 ∈ 상품별 허용 목록 |
| 전용면적 | 면적 ≤ 85㎡ (정부상품) 또는 제한없음 (금융권) |
| 중복대출없음 | 기존 전세대출 보유 = N |
| 정보완전성 | 필수정보 전 항목 수집 = Y |
| 상품별 필수 | A3: 혼인 7년 이내 / A4: 출산 2년 이내 / A2·A5: 만 19~34세 |

**실행 결과 (500명 기준)**

| 항목 | 값 |
|------|-----|
| 고객 × 상품 조합 | 30,000행 |
| Gate 통과 조합 | 13,952행 (46.5%) |
| 1개 이상 통과 고객 | 257명 / 500명 (51.4%) |
| 추천 불가 고객 | 243명 |

추천 불가 주요 사유: 세대주 아님(83명) · 무주택 아님(23명) · 기존 전세대출 중복 · 신용점수 미달

### 핵심 가중치 항목·비중 — JeonseLoanRecScore

```
JeonseLoanRecScore = JeonseLoanGate × (
      w_A · ApprovalScore            ← 승인 가능성
    + w_I · InterestBenefitScore     ← 금리 유리도
    + w_L · LimitCoverageScore       ← 한도 커버율
    + w_F · AffordabilityScore       ← 상환 여력
    + w_G · GuaranteeStabilityScore  ← 보증 안정성
    + w_P · PreferenceMatchScore     ← 우대조건 충족도
    − RiskPenalty
)
```

#### 고객 유형별 가중치

고객 상황에 따라 중요한 가치가 다르므로 **7개 유형 × 개별 가중치**로 설계.
여러 유형 동시 해당 시 실패 비용이 큰 보호형 우선 적용 (신용취약 > 상환위험 > 한도부족 > …).

| 고객 유형 | 판별 조건 | Approval | Interest | Limit | Afford | Guarantee | Pref |
|-----------|----------|:--------:|:--------:|:-----:|:------:|:---------:|:----:|
| 기본형 | 해당 없음 | 0.25 | 0.25 | 0.20 | 0.15 | 0.10 | 0.05 |
| **금리민감형** | 연소득 ≤ 4,000만원 | 0.20 | **0.38** | 0.18 | 0.14 | 0.05 | 0.05 |
| **한도부족형** | 신청금 > 최대한도 90% | 0.20 | 0.15 | **0.38** | 0.15 | 0.07 | 0.05 |
| **신용취약형** | 신용점수 600~699 | **0.40** | 0.20 | 0.18 | 0.12 | 0.05 | 0.05 |
| 청년무주택형 | 만 19~34세 + 무주택 세대주 | 0.20 | **0.35** | 0.20 | 0.10 | 0.10 | 0.05 |
| 신혼·신생아형 | 혼인 7년 이내 or 출산 2년 이내 | 0.22 | **0.30** | 0.25 | 0.13 | 0.05 | 0.05 |
| **상환위험형** | 예상 DSR ≥ 35% | 0.25 | 0.18 | 0.15 | **0.32** | 0.05 | 0.05 |

#### RiskPenalty (감점)

| 조건 | 감점 |
|------|------|
| DSR 30~40% 구간 | −10점 |
| 신용점수 600~699점 | −15점 |
| 보증금/시세 > 90% (깡통전세) | −20점 |
| 선순위채권+보증금 > 시세 80% | −10점 |

## ③ 어려웠던 부분 / 막힌 점

### 1. 단위 불일치 — 만원 vs 원

소득·보증금·부채는 **만원** 단위, 일부 계산식은 **원** 단위가 섞여 들어오면서
DSR 계산 결과가 100배 오차나는 버그가 발생. 모든 금액 컬럼을 만원 단위로 통일하는 전처리 규칙을 명시적으로 고정해 해결.

### 2. 가중치 근거 — 팀 내 의견 충돌

"금리 vs 승인 어느 쪽이 더 중요한가"를 두고 팀 안에서 의견이 갈림.
결론: 전세대출은 **계약 후 신청** 구조이므로 실패 비용이 가장 크다 → 승인 0.25 · 금리 0.25 동등 최우선으로 합의. 이후 고객 유형별로 비중을 차별화하는 방식으로 절충.

### 3. Colab에서 Streamlit 띄우기

`streamlit run`이 Colab 셀에서 바로 실행되지 않아 백그라운드 프로세스 + cloudflared 터널 방식으로 우회. 초기 설정에 시간이 걸리고 터널 URL이 매 실행마다 바뀌는 불편함이 있었음.

### 4. FSS API 파일명 공백·괄호 문제

금감원 API에서 내려받은 CSV 파일명에 공백과 괄호(`fss_jeonse_merged_20260630 (1).csv`)가 포함되어 `glob` 패턴으로 찾지 못하는 `ValueError: NoneType` 오류 발생. fallback glob 패턴 다단계 탐색으로 해결.

### 5. 상품 중복 노출

`merge_jeonse_product_catalog.csv`에 정부지원(A1~A5)과 동일 은행 상품(B 계열)이 함께 있어, FSS 데이터와 concat하면 같은 은행 상품이 "정부지원"과 "은행" 두 번 노출되는 문제가 발생. 카탈로그에서 A 상품만 필터링한 뒤 FSS 데이터를 붙이는 방식으로 해결.

### 6. InterestBenefitScore 정규화 범위

초기에는 전체 상품 기준으로 금리를 정규화해 고객마다 후보군이 달라도 동일 점수가 매겨지는 문제가 있었음. Gate 통과 상품을 **고객별로 묶어** 그 안에서 min-max 정규화하는 방식으로 수정.
